## Summary

Eddie Lazer, Head of Engineering at a16z Crypto, argues that token design is
fundamentally protocol design, where tokens act as tools to achieve specific
decentralized goals. He outlines a structured framework for development—moving
from goals and constraints to mechanisms—and provides a comprehensive taxonomy
of token capabilities across five key categories.

## Key Points

- Tokens are tools that serve protocols, not the primary object of design.
- The protocol design process should flow from defining a measurable Goal to
  identifying Constraints, and finally selecting Mechanisms.
- Constraints are divided into endogenous (chosen trade-offs) and exogenous
  (imposed by tech or regulation).
- Common failure patterns include 'token-first thinking' and using the community
  as a 'wastebasket' for unresolved design problems.
- Token capabilities are categorized into five groups: Payment, Voting,
  Stakeholding, Metadata, and Claiming.
- Advanced token metadata can include embedded interfaces (HTML/JS) so the token
  carries its own UI.
