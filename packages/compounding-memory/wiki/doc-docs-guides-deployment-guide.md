# The New Fuse Platform - Complete Deployment Guide

## Overview

This guide implements a three-phase deployment strategy for The New Fuse
Platform, prioritizing speed-to-market initially, then progressively optimizing
for cost-efficiency and granular control as we scale.

## Architecture Summary

**Tech Stack:**

- **Frontend**: React 18.2 + Vite + TypeScript
- **Backend**: NestJS 11.0+ with Bun runtime
- **Database**: PostgreSQL with Drizzle ORM
- **Cache**: Redis
- **AI Integration**: MCP (Model Context Protocol) + Multiple AI providers
- **Build System**: Turborepo with memory-optimized builds
- **Containerization**: Docker with multi-stage builds

## Operational Runbook (Current Scripts)

In addition to the phased strategy below, day-to-day production deployments use
the automation scripts in `scripts/deployment`.

### Standard Automated Flow

```bash
# 1. Validate environment and config
./scripts/deployment/validate-deployment.sh

# 2. Deploy with automation + rollback hooks
./scripts/deployment/deploy-automated.sh

# 3. Verify deployment health
./scripts/deployment/smoke-tests.sh
```

### CloudRuntime-Focused Script Flow

```bash
# Deploy all services
./scripts/deployment/cloud_runtime-deploy.sh all

# Deploy one service
./scripts/deployment/cloud_runtime-deploy.sh deploy api-gateway

# Stream service logs
./scripts/deployment/cloud_runtime-deploy.sh watch api-gateway
```

### Manual Controlled Flow

```bash
./scripts/deployment/validate-deployment.sh
./scripts/deployment/db-backup.sh
./scripts/deployment/db-migrate.sh
pnpm run build:cloud_runtime
cloud_runtime up --service api-gateway
cloud_runtime up --service backend
cloud_runtime up --service frontend
./scripts/deployment/smoke-tests.sh
```

### Deployment Strategy Toggles

Set `.deployment-config` as needed:

- `DEPLOYMENT_STRATEGY=rolling|blue-green|canary`
- `AUTO_ROLLBACK=true|false`
- `BACKUP_BEFORE_MIGRATE=true|false`

## Phase 1: Short-Range Plan (Managed PaaS) 🚀

**Goal**: Achieve production deployment within days using fully managed
platforms.

### Platforms

- **Frontend**: Vercel
- **Backend Services & Databases**: CloudRuntime

### Quick Start

```bash
# 1. Clone and setup
git clone <repository>
cd the-new-fuse
pnpm install

# 2. Configure environment
cp .env.example .env.production
# Fill in all required secrets

# 3. Deploy everything
./deployment/phase1-deploy.sh
```

### Manual Deployment Steps

#### Frontend (Vercel)

1. **Connect Repository**:
   - Link GitHub repository to Vercel
   - Set root directory: `apps/frontend`
   - Build command: `pnpm run build`
   - Output directory: `dist`

2. **Environment Variables**:
   ```bash
   VITE_API_URL=https://your-cloud_runtime-api-url.thenewfuse.com
   VITE_WS_URL=wss://your-cloud_runtime-api-url.thenewfuse.com
   NODE_ENV=production
   ```

#### Backend (CloudRuntime)

1. **Database Setup**:
   - Create PostgreSQL service
   - Create Redis service
   - Note connection URLs

2. **Deploy Services**:

   ```bash
   cloud_runtime login
   cloud_runtime up
   ```

3. **Configure Environment Variables**:

   ```bash
   # Auto-generated by CloudRuntime
   DATABASE_URL=${{Postgres.DATABASE_URL}}
   REDIS_URL=${{Redis.REDIS_URL}}

   # Manual configuration
   JWT_SECRET=your-jwt-secret
   OPENAI_API_KEY=your-openai-key
   ANTHROPIC_API_KEY=your-anthropic-key
   AWS_ACCESS_KEY_ID=your-aws-key
   AWS_SECRET_ACCESS_KEY=your-aws-secret
   ```

## Phase 2: Mid-Range Plan (Google Cloud Run) ⚡

**Goal**: Optimize high-traffic services for cost (scale-to-zero) and
performance.

### Target Services

- AI/ML services (MCP Core)
- A2A Service (spiky workloads)
- Sync Core (resource-intensive)

### Setup

```bash
# 1. Setup Google Cloud
./deployment/phase2-gcr-setup.sh

# 2. Configure GitHub Secrets
# Add GCP_SERVICE_ACCOUNT_KEY to repository secrets

# 3. Push changes to trigger deployment
git push origin main
```

### Features

- **Serverless**: Scale to zero when not in use
- **Auto-scaling**: Handle traffic spikes automatically
- **Cost Optimization**: Pay only for actual usage
- **CI/CD**: Automated deployments via GitHub Actions

### Migration Strategy

1. Start with MCP Core (AI workloads)
2. Move A2A Service (communication spikes)
3. Migrate Sync Core (resource optimization)
4. Keep API Server on CloudRuntime (consistent load)

## Phase 3: Long-Range Plan (Kubernetes/GKE) 🏗️

**Goal**: Maximum control and orchestration for enterprise scale.

### Setup

```bash
# 1. Setup GKE cluster and components
./deployment/phase3-k8s-setup.sh

# 2. Deploy using Helm
helm upgrade --install the-new-fuse ./deployment/kubernetes/helm/the-new-fuse \
  --namespace the-new-fuse \
  --create-namespace \
  --set global.environment=production
```

### Features

- **Full Control**: Complete infrastructure management
- **Advanced Orchestration**: Complex deployment patterns
- **Enterprise Security**: Network policies, RBAC, encryption
- **Observability**: Prometheus, Grafana, distributed tracing
- **Multi-tenancy**: Namespace isolation, resource quotas

### Components Installed

- **GKE Cluster**: Auto-scaling node pools
- **NGINX Ingress**: Load balancing and SSL termination
- **cert-manager**: Automatic TLS certificate management
- **External Secrets**: Secure secret management
- **Prometheus Stack**: Comprehensive monitoring
- **Workload Identity**: Secure GCP integration

## Multi-Tenant Configuration

### Phase 1 (Environment Variables)


```bash

```

## Troubleshooting

### Common Issues

#### Build Failures

```bash
# Check Turbo configuration
pnpm run build --filter=@the-new-fuse/api-server

# Clear cache
pnpm run clean:cache
pnpm install --frozen-lockfile
```

#### Database Connection Issues

```bash
# Generate Drizzle client
pnpm run db:generate

# Run migrations
pnpm run db:migrate
```

#### Service Communication

```bash
# Test health endpoints
curl https://api-server.thenewfuse.com/health

# Check logs
cloud_runtime logs --service api-server
```
