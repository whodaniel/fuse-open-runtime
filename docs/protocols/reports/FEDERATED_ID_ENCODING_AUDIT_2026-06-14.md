## 1. What "Federated ID" actually means in TNF

After the consistency review (Phase 8) the `role`/`qualities`/`platform` axes
were aligned. This audit takes up the next layer: the three federated-ID
namespaces that correlate an agent identity across subsystems. Concretely:

- `canonicalEntityId` is the agent's **hierarchical** identifier. Format
  `TNF:[scope:]CATEGORY:PROVIDER:NAME:INSTANCE`, validated by
  `normalizeCanonicalEntityId()`. Used by the relay registry, master-clock, and
  broker dispatch.
- `idNumber` is the agent's **federated reputation sequence** — `ID#:<Base58>`
  generated by `FederatedIdentityService.generateIdNumber()` (sequential Redis
  INCR + Base58 encode). Used for crypto-attribution of AI Wiki entries,
  compounding memory, and transcript stovepipes.
- `mcid` (`tnf/mcid/0.1`) is the **UUID lineage envelope** that links TWIP,
  handoff, relay, and workflow events. `correlation_id`, `causation_id`,
  `trace_id` are all UUID v4.

Before Phase 9 these three namespaces had partial or broken coverage:

- `canonicalEntityId` was carried only inside `agents.infoRecord`; the `agents`
  DB table had no column for it.
- `idNumber` was never persisted on agents — only generated ad-hoc inside
  `FederatedIdentityService` and consumed by the Wiki stovepipe.
- The agent-registry-bridge emitted `canonicalEntityId: "AGENT://TNFCORE/..."`
  which fails `normalizeCanonicalEntityId()` (segments[0] must be `TNF`).
- `TranscriptProcessorV2/V3/V4` re-implemented the Base58 alphabet inline
  instead of using `FederatedIdentityService`.
- `gemini-browser-skill/TranscriptProcessorV2.ts` had the canonical import
  commented out
  (`// import { FederatedIdentityService } from '@the-new-fuse/a2a-core'`).

Phase 9 fixes all of this.

## 2. Concrete changes

### 2.1 agent-registry-bridge (`packages/relay-core/src/agent-registry-bridge.ts`)

- Imports `buildCanonicalEntityId` from `./contracts/identity.js`.
- Replaces the malformed `AGENT://TNFCORE/${AGENT_ID}` with
  `buildCanonicalEntityId({ category: 'AGENT', provider: 'TNFCORE', name: AGENT_ID, instance: 1 })`.
- Now produces values like `TNF:AGENT:TNFCORE:launchpad-agent:001`.

### 2.2 Database schema (`packages/database/src/drizzle/schema/agents.ts`)

- Adds `canonicalEntityId` (varchar 255) — TNF-namespaced hierarchical ID.
- Adds `idNumber` (varchar 64) — `ID#:<Base58>` federated reputation sequence.
- Adds `federation` (jsonb) — umbrella column carrying all three federated
  namespaces as a single bundle for hot-path correlation queries.
- Migration `0010_add_federated_ids.sql` creates columns + indexes:
  - Unique partial index on `id_number` when not null.
  - GIN index on `federation`.
  - B-tree indexes on `federation->>'canonicalEntityId'` and
    `federation->>'idNumber'`.

### 2.3 Seeder (`packages/database/scripts/seed-agent-registry.ts`)

- Adds deterministic `canonical_entity_id` and `id_number` per seed row.
- `id_number` is derived from `FNV-1a(name) + 1000` then encoded to Base58. This
  gives stable values across re-runs while remaining visually distinct from
  production sequential IDs (which start at 1 and are managed by
  `FederatedIdentityService`).
- `federation` jsonb populated per row with `scopes` derived from
  `qualities.orchestrates_agents` (`'dacc-orchestrator'` vs `'dacc-worker'`).

### 2.4 Transcript Processor V2/V3/V4 (`packages/gemini-browser-skill/src/*.ts`)

- V2: exports a new top-level helper:
  ```ts
  export const FEDERATED_BASE58_ALPHABET = '...';
  export function generateFederatedIdNumber(seq: number): string { ... }
  ```
  Replacing the inline alphabet + loop with a single function call
  `generateFederatedIdNumber(video.index)`.
- V3 and V4: import the helper from `./TranscriptProcessorV2.js`. Behavior is
  preserved for `seq = 0` (returns `ID#:1`, the alphabet's first character) so
  existing transcript IDs that emit `ID#:1` keep emitting `ID#:1`.

### 2.5 Documentation (`.agent/ROLE_DEFINITIONS.md`)

- New section **🌐 Federated ID Encoding (Phase 9)** documents the three
  namespaces, encoding rules, source-of-truth files, and the relationship to
  other identifiers (`MASTER_LIBRARY_ID`/`vector_id`, `ID#:PROT-HANDOFF-...`,
  `AGENT-XX` runtime handles).

## 3. What's still fragmented (open follow-up)

### 3.1 The `ID#:` prefix is shared with `vector_id`

`scripts/autonomy/generate_merkle_tree.py` uses
`ID#:${base58.b58encode(hash[:8])}` for `KNOWLEDGE_TREE.json` leaf vectors.
These are hash-derived **intelligence-vector IDs**, not federated reputation
sequences. The shared `ID#:` prefix is documented as a conflict in
`ROLE_DEFINITIONS.md` (Phase 9 section) but no migration has been attempted.
Three options exist:

(a) leave the prefix and rely on context/auditability; (b) replace `ID#:` with
`VEC#:` on the next knowledge-tree rebuild; (c) replace `ID#:` with `FED#:` for
the federated reputation sequence.

The audit recommends **(a) for now** — both encodings carry semantic distinction
that downstream consumers already key off (sequence position vs hash hex
prefix). A breaking change risks regression in unrelated tooling.

### 3.2 `mcid` bundle is JSON-encoded only

The `mcid` schema documents a structured envelope; in `agents.federation` it is
nullable. Once agents start claiming lineage at registration time (it currently
happens at message-dispatch time), this column should become populated by the
master-clock envelope path.

### 3.3 Production-built IDs do not flow back to the registry

`FederatedIdentityService` increments `tnf:identity:seq:<agentId>` on demand but
does not write back to the `agents.id_number` column. The seeder mirrors the
sequence algebra so seeded entries get plausible values, but a live
`tnf register` call does not currently SELECT-then-UPDATE the agents row. A
follow-up should wire this so the registry and the FederatedIdentityService
counters stay in sync.

### 3.4 The bridge still emits `operationalHandle = AGENT_ID`

The bridge uses `process.env.AGENT_ID || 'LAUNCHPAD-AGENT'` as the operational
handle. Per `ROLE_DEFINITIONS.md` the handle format is `AGENT-XX`. This is
acceptable for the current LAUNCHPAD flow but should be validated when more
upstream emitters register.

## 5. Memory write

Saving the rule: when auditing federated identity, the alphabet is the
benchmark, not the prefix. If a downstream consumer uses Base58 with the
canonical alphabet, it's safe to assume equivalence with the canonical encoder
even if the import was bypassed. If the alphabet differs in even one character,
IDs are no longer comparable across systems — that's a silent data corruption
risk. The transitive coupling between `FederatedIdentityService.ALPHABET` and
the `FEDERATED_BASE58_ALPHABET` constant in `TranscriptProcessorV2.ts` is now
documented in both files.
