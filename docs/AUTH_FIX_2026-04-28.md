# Auth System Fix — 2026-04-28

## Summary

Register and login were completely broken for approximately one week. All three
auth paths (Supabase sign-in, direct API login, and direct API registration)
failed silently or with opaque errors. This document records the root causes,
the fixes applied, and the verification steps taken.

## Root Causes

### Bug 1 — Cloudflare Turnstile Mismatch (Critical)

The API gateway (`api-gateway` Cloud Run service) had `AUTH_REQUIRE_TURNSTILE=true`
in its environment, causing every `POST /v1/auth/login` and
`POST /v1/auth/register` request to reject with 401
"Cloudflare Turnstile token is required" when no Turnstile token was provided.

The frontend production build (`.env.production`) did **not** include
`VITE_TURNSTILE_SITE_KEY` or `VITE_AUTH_REQUIRE_TURNSTILE`. The Turnstile
widget component (`TurnstileWidget.tsx`) only renders when **both** are truthy:

```tsx
{requireTurnstile && turnstileSiteKey && (
  <TurnstileWidget siteKey={turnstileSiteKey} ... />
)}
```

Since `VITE_AUTH_REQUIRE_TURNSTILE` was falsy at build time, the widget was
never rendered, no token was ever generated, and every direct API login and
registration attempt returned 401.

The Supabase-first path (`useAuth.tsx` lines 254-281) was also affected: when
Supabase sign-in succeeded but the subsequent token exchange failed (see Bug 2),
the fallback to the direct API path also hit the Turnstile wall.

**Impact:** Every direct API login and registration returned 401. The Supabase
path also failed due to this + Bug 2.

### Bug 2 — Missing Supabase Environment Variables on API Gateway (Critical)

The `api-gateway` Cloud Run service was deployed without `SUPABASE_URL` or
`SUPABASE_ANON_KEY` environment variables. These are required by the Supabase
token exchange endpoint (`POST /v1/auth/supabase`) in
`gateway-auth.service.ts:404-445`.

The service's `verifySupabaseAccessToken()` method checks for these vars and
throws `UnauthorizedException('Supabase auth is not configured (missing
SUPABASE_URL or SUPABASE_ANON_KEY)')` when they are absent.

**Impact:** Even when Supabase sign-in succeeded client-side, the backend
token exchange that converts the Supabase access token into a TNF JWT failed
with 401, making the primary auth path completely dead.

### Bug 3 — 3-Second Request Timeout (High)

`useAuth.tsx` line 12 set `REQUEST_TIMEOUT_MS = 3_000`. Cloud Run cold starts
typically take 5-10 seconds. The first auth request after an idle period would
abort via `AbortController`, surfacing as a generic network error to the user
with no indication that a retry would succeed.

**Impact:** First login attempt after idle period always timed out.

## Fixes Applied

### Fix 1 — Enable Turnstile Consistently on Frontend

**File:** `apps/frontend/.env.production`

Added two environment variables that were present in `.env.local` but missing
from the production build config:

```
VITE_TURNSTILE_SITE_KEY=0x4AAAAAAChQ2ZeB10ay0LvR
VITE_AUTH_REQUIRE_TURNSTILE=true
```

These are baked into the Vite build at compile time via `import.meta.env`. The
Turnstile widget now renders on the login and register pages in production,
generating a `cfTurnstileToken` that is passed to the API gateway.

**Note:** The corresponding `CLOUDFLARE_TURNSTILE_SECRET_KEY` was already set
on the API gateway Cloud Run service (added during this fix session).

### Fix 2 — Add Supabase Env Vars to API Gateway Cloud Run

**Command:**

```bash
gcloud run services update api-gateway \
  --region us-central1 \
  --update-env-vars \
    "AUTH_REQUIRE_TURNSTILE=true,\
     CLOUDFLARE_TURNSTILE_SECRET_KEY=0x4AAAAAAChQ2aTnwtUyd4q-q8RkP8QNjnc,\
     SUPABASE_URL=https://wslydgtgindrywldatbv.supabase.co,\
     SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

This deployed a new Cloud Run revision: `api-gateway-00029-dnb`.

The Supabase exchange endpoint now properly validates Supabase access tokens
against `https://wslydgtgindrywldatbv.supabase.co/auth/v1/user` and issues
TNF JWTs in return.

### Fix 3 — Increase Auth Request Timeout

**File:** `apps/frontend/src/hooks/useAuth.tsx`

```diff
- const REQUEST_TIMEOUT_MS = 3_000;
+ const REQUEST_TIMEOUT_MS = 15_000;
```

15 seconds is sufficient for Cloud Run cold starts while still preventing
indefinite hangs.

### Fix 4 — Google OAuth Button Visibility

**Files:**
- `apps/frontend/src/pages/auth/Login.tsx`
- `apps/frontend/src/pages/auth/Register.tsx`

The "Continue with Google" button used `text-slate-900` (near-black) on
`bg-transparent` against a `bg-slate-950` page background, making it invisible.

```diff
- className="... text-slate-900 hover:bg-slate-100 ..."
+ className="... text-white hover:bg-slate-800 ..."
```

## Auth Flow After Fix

### Path A: Supabase Sign-In (Primary)

1. User enters email + password on Login page
2. Frontend calls `supabase.auth.signInWithPassword()` directly against Supabase
3. Supabase returns an `access_token`
4. Frontend sends `POST /v1/auth/supabase` with `{ accessToken }` to the API gateway
5. API gateway validates the Supabase token against
   `https://wslydgtgindrywldatbv.supabase.co/auth/v1/user`
6. API gateway looks up (or auto-creates) the user in the `users` table
7. API gateway issues a TNF JWT (15m access + 7d refresh)
8. Frontend stores the TNF JWT in `localStorage` under key `auth_token`

### Path B: Direct API Login (Fallback)

1. User enters email + password on Login page
2. Turnstile widget renders and generates a `cfTurnstileToken`
3. Frontend sends `POST /v1/auth/login` with
   `{ email, password, cfTurnstileToken }`
4. API gateway verifies the Turnstile token with Cloudflare
5. API gateway verifies the password with `bcrypt.compare`
6. API gateway issues TNF JWT tokens
7. Frontend stores the TNF JWT

### Path C: Direct API Registration

1. Same as Path B but sends `POST /v1/auth/register` with
   `{ name, email, password, cfTurnstileToken }`
2. API gateway verifies Turnstile, checks for existing user, hashes the
   password, creates the user in the `users` table, and issues tokens

## Security Advisory

The following issues were identified during this fix but are **not yet resolved**:

1. **Weak JWT_SECRET**: The `api-gateway` Cloud Run service uses
   `JWT_SECRET=the-new-fuse-2025-secret-key-change-me`. This should be rotated
   to a cryptographically random 32+ character string. Changing it will
   invalidate all active sessions.

2. **Secrets committed to repository**: The files `cloud_runtime-vars-api-gateway.txt`,
   `cloud_runtime-vars-backend.txt`, `cloud_runtime-vars-frontend.txt`, and
   `cloud_runtime-vars-relay.txt` contain production secrets in plaintext (JWT_SECRET,
   CLOUDFLARE_TURNSTILE_SECRET_KEY, SUPABASE_SERVICE_ROLE_KEY, DATABASE_URL
   credentials). These files should be added to `.gitignore` and all exposed
   secrets should be rotated.

3. **Token blacklist in-memory only**: `apps/backend/src/auth/token-blacklist.service.ts`
   uses an in-memory `Map`. Blacklisted tokens are lost on restart or across
   instances. Should use Redis (Upstash is already configured).

4. **No refresh token rotation**: The api-gateway issues refresh tokens but does
   not store or validate them against the database. A stolen refresh token
   cannot be revoked.
